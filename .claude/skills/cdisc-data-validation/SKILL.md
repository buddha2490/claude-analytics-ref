# CDISC Data Validation Skill

Validates SDTM and ADaM datasets against CDISC standards. Takes an HTML data profile report (generated by the `profile-data` skill) as input, re-loads the underlying XPT/RDS files using the same R helper functions that produced the report, checks variable names, types, labels, controlled terminology values, and missing value handling — then produces a Quarto validation report with findings for each domain.

## Trigger

Use this skill when the user asks to:
- Validate data against CDISC standards
- Check CDISC compliance of datasets
- Evaluate SDTM or ADaM domains for standards adherence
- Review variable types, labels, or values against CDISC
- Identify missing required variables in SDTM/ADaM datasets
- Run a CDISC data quality report

## Usage

```bash
/cdisc-data-validation <path-to-data-profile-report.html>
/cdisc-data-validation projects/exelixis-sap/output-reports/exelixis-dummy-data-report.html
/cdisc-data-validation projects/exelixis-sap/output-reports/exelixis-dummy-data-report.html --domains=DM,AE,EX
```

## Parameters

- **First argument** (required): Path to the HTML data profile report generated by the `profile-data` skill. The data directories are inferred from this path.
- `--domains` (optional): Comma-separated list of domains to validate. Default: all domains found.
- `--output` (optional): Output path for the validation QMD. Default: same directory as input, named `cdisc-validation-report-<date>.qmd`.

## Process

### Step 1 — Build the Dataset Inventory via R

Run an R script (using the `r-clinical-programmer` agent or inline) that sources the existing helper functions and builds a structured inventory of every domain.

```r
library(haven)
library(dplyr)
library(stringr)
library(here)

# Source the existing helper functions
source(here("R/summarize_numeric.R"))
source(here("R/summarize_character.R"))
source(here("R/summarize_date.R"))
source(here("R/summarize_variable.R"))
source(here("R/extract_dataset_metadata.R"))

# --- Load SDTM XPT files ---
sdtm_dir <- here("projects/exelixis-sap/output-data/sdtm")
sdtm_files <- list.files(sdtm_dir, pattern = "\\.xpt$", full.names = TRUE)

sdtm_datasets <- list()
for (file in sdtm_files) {
  domain <- toupper(tools::file_path_sans_ext(basename(file)))
  sdtm_datasets[[domain]] <- read_xpt(file)
}

# --- Load ADaM RDS files ---
adam_dir <- here("projects/exelixis-sap/output-data/adam")
adam_files <- list.files(adam_dir, pattern = "\\.rds$", full.names = TRUE)

adam_datasets <- list()
for (file in adam_files) {
  domain <- toupper(tools::file_path_sans_ext(basename(file)))
  adam_datasets[[domain]] <- readRDS(file)
}

# --- Build inventory for each domain ---
# For each dataset, extract:
#   1. metadata:   extract_dataset_metadata(dataset) → Variable, Type, Length, Format, Label
#   2. summaries:  for each variable, summarize_variable(dataset[[var]], var)
#                  - character vars → frequency table: Value | N | (%)
#                  - numeric vars   → stats table:     N | Mean | SD | Median | Min | Max | Missing
#                  - date vars      → range table:     Earliest Date | Latest Date | Range (days) | N Non-Missing | Missing

build_inventory <- function(datasets) {
  lapply(names(datasets), function(domain) {
    ds <- datasets[[domain]]
    list(
      domain   = domain,
      records  = nrow(ds),
      n_vars   = ncol(ds),
      metadata = extract_dataset_metadata(ds),
      summaries = setNames(
        lapply(names(ds), function(v) summarize_variable(ds[[v]], v)),
        names(ds)
      )
    )
  }) |> setNames(names(datasets))
}

sdtm_inventory <- build_inventory(sdtm_datasets)
adam_inventory  <- build_inventory(adam_datasets)
```

The inventory gives Claude everything needed for CDISC validation without re-reading the HTML:
- **Variable names, types, labels** (from `metadata`)
- **Unique values with frequencies** for character vars (from `summaries` — used for CT checks)
- **Missing counts** for all variable types (from `summaries`)
- **Date ranges** for date variables (from `summaries`)

### Step 2 — Query CDISC Standards via RAG

For each domain in the inventory, call `mcp__npm-rag-v1__query_documents`. Run domain-specific and cross-domain queries:

**Domain-specific queries:**

| Domain | Query |
|--------|-------|
| DM | `"SDTM DM demographics domain required variables USUBJID RFSTDTC SEX RACE ETHNIC"` |
| AE | `"SDTM AE adverse events required variables AETERM AEDECOD AESEV AESER AEREL AESEQ"` |
| EX | `"SDTM EX exposure required variables EXTRT EXDOSE EXDOSEU EXSTDTC EXROUTE"` |
| LB | `"SDTM LB laboratory results required variables LBTESTCD LBTEST LBSTRESN LBSTRESU"` |
| VS | `"SDTM VS vital signs required variables VSTESTCD VSTEST VSSTRESN VSSTRESU"` |
| CM | `"SDTM CM concomitant medications required variables CMTRT CMDECOD CMSTDTC"` |
| MH | `"SDTM MH medical history required variables MHTERM MHDECOD MHSTDTC MHCAT"` |
| DS | `"SDTM DS disposition required variables DSDECOD DSTERM DSSTDTC DSCAT"` |
| IE | `"SDTM IE inclusion exclusion criteria required variables IETESTCD IETEST IEORRES"` |
| RS | `"SDTM RS response domain required variables RSTEST RSORRES RSSTRESC"` |
| TU | `"SDTM TU tumor identification required variables TUTESTCD TUTEST TUORRES"` |
| TR | `"SDTM TR tumor results required variables TRTESTCD TRTEST TRORRES TRSTRESN"` |
| QS | `"SDTM QS questionnaire required variables QSTESTCD QSTEST QSORRES QSSTRESC"` |
| ADSL | `"ADaM ADSL subject level analysis dataset required variables SAFFL ITTFL TRTP TRTPN"` |
| ADAE | `"ADaM ADAE adverse events analysis dataset required variables AETERM AEDECOD TRTEMFL"` |
| ADTTE | `"ADaM ADTTE time to event analysis dataset required variables AVAL PARAM CNSR EVNTDESC"` |
| ADRS | `"ADaM ADRS response analysis dataset required variables AVAL PARAM AVALC AVISITN"` |

**Cross-domain queries (run once):**

- `"CDISC controlled terminology SEX RACE ETHNIC codelist values"` — DM CT
- `"CDISC controlled terminology AESEV AESER AEREL adverse event"` — AE CT
- `"CDISC controlled terminology EXROUTE exposure route of administration"` — EX CT
- `"CDISC SDTM variable naming convention uppercase 8 characters domain prefix suffix"` — naming rules
- `"SDTM date format ISO 8601 DTC variables YYYY-MM-DD partial dates"` — date rules
- `"ADaM analysis flags Y empty string missing value handling SAFFL ITTFL population flag"` — flag coding
- `"CDISC SDTM required variables all domains STUDYID DOMAIN USUBJID SEQ"` — universal required vars

### Step 3 — Validate Each Domain

Apply the following checks for each domain using the inventory (Step 1) and RAG results (Step 2):

#### 3a. Required Variable Check
- Compare variable names in `metadata$Variable` against CDISC-required variables from RAG.
- **BLOCKING**: Required variable absent from the dataset.
- **NOTE**: Extra non-standard variables — flag those with empty labels.

#### 3b. Variable Naming Conventions
For every name in `metadata$Variable`:
- [ ] All uppercase
- [ ] ≤8 characters
- [ ] Domain prefix correct (e.g., `AE` prefix in the AE domain)
- [ ] Suffix conventions: `--DTC` dates, `--TERM` verbatim, `--DECOD` decoded, `--CD` codes, `--N` numeric counterpart
- **WARNING**: Any naming violation.

#### 3c. Variable Type Check
Using `metadata$Type` (values: `Num`, `Char`, `Date`, `DateTime`, `Factor`):
- `--DTC` variables → must be `Char`, not `Date`/`DateTime`/`Num`
- `--STRESN`, `AVAL`, dose variables → must be `Num`
- Flag variables (`SAFFL`, `ITTFL`, etc.) → must be `Char`
- `--SEQ` variables → must be `Num`
- **WARNING**: Type mismatch.

#### 3d. Variable Label Check
Using `metadata$Label`:
- **WARNING**: Empty string label — variable lacks a label (required for XPT transport).
- **NOTE**: Label >200 characters.

#### 3e. Controlled Terminology Check
For character variables with CDISC codelists, the `summaries` frequency table (Value | N | (%)) gives all unique observed values. Compare against the RAG codelist:
- **BLOCKING**: Value not in a **closed** CDISC codelist.
- **NOTE**: Value in an **extensible** codelist that is not a standard CDISC term (may be a valid sponsor extension — note and flag for human review).

Key variables to check: `SEX`, `RACE`, `ETHNIC`, `AESEV`, `AESER`, `AEREL`, `EXROUTE`, `VSSTRESU`, `LBSTRESU`, `DSDECOD`, `DSTERM`.

For ADaM flag variables (`SAFFL`, `ITTFL`, `TRTEMFL`, etc.), check `summaries` frequency values:
- Valid: `"Y"` and `""` (empty string) only
- **WARNING**: `"TRUE"` / `"FALSE"` / `NA` values present — flag variables must use `"Y"` / `""` encoding.

#### 3f. Date Format Check
For variables with `--DTC` suffix, the `summaries` result determines how to check:
- If the variable produced a **date range table** (Earliest Date | Latest Date | Range (days)): dates parsed successfully — ISO 8601 format is valid. Note any `Missing` counts.
- If the variable produced a **frequency table** (Value | N | (%)): dates were NOT parsed as dates. Inspect the `Value` column for non-ISO 8601 patterns.
- **BLOCKING**: Any value that doesn't match `YYYY-MM-DD` or `YYYY-MM-DDThh:mm:ss`.
- **NOTE**: Partial dates observed (`YYYY-MM` or `YYYY`).

#### 3g. Missing Value Handling
- **Numeric variables**: Check `Missing` column in the stats table. Flag required variables with `Missing > 0` as **BLOCKING**.
- **Character variables**: Check for `(Missing)` row in frequency table. Flag required variables with any `(Missing)` as **BLOCKING**.
- **`USUBJID`**: Must have zero missing values in every domain — **BLOCKING** if any.
- **High missingness**: Non-required variables with >20% missing → **WARNING**.

#### 3h. Cross-Domain Consistency
The R inventory contains full datasets. Check:
- All `USUBJID` in non-DM domains must exist in DM: `anti_join(domain_ds, dm_ds, by = "USUBJID")` — **BLOCKING** if non-empty.
- `STUDYID` must be identical across all domains — **BLOCKING** if inconsistent.
- Event dates within DM `RFSTDTC`–`RFENDTC` range (where both present) — **WARNING** if violations found.

### Step 4 — Classify Findings

| Severity | Meaning |
|----------|---------|
| **BLOCKING** | Missing required variable, invalid CT value in closed codelist, malformed date, USUBJID missing or cross-domain mismatch |
| **WARNING** | Missing label, type mismatch, flag encoded as TRUE/FALSE, high missingness on non-required variable |
| **NOTE** | Naming edge case, extensible CT value, partial date observed, informational |
| **PASS** | Check passed with no issues |

### Step 5 — Generate Quarto Report

Write a static `.qmd` to the same directory as the input HTML, named `cdisc-validation-report-<YYYY-MM-DD>.qmd`.

**YAML header:**
```yaml
---
title: "CDISC Data Validation Report"
subtitle: "<Study Name> — SDTM and ADaM Datasets"
author: "Automated Validation"
date: "<today>"
format:
  html:
    toc: true
    toc-depth: 3
    toc-location: left
    number-sections: true
    theme: cosmo
    embed-resources: true
---
```

**Report sections:**

1. **Abstract** — 3–5 sentences: datasets analyzed, total BLOCKING/WARNING/NOTE counts, headline issues, overall compliance posture.

2. **Source Report** — Path to the input HTML profile report and the date it was generated.

3. **Summary of Required Changes** — Numbered list grouped by theme (missing variables, type errors, missing labels, CT violations, date formats, missing values), ordered by severity. Each item names the domain(s) and required fix.

4. **Overall Statistics Table:**

| Domain | Records | Variables | Blocking | Warnings | Notes | Status |
|--------|---------|-----------|----------|----------|-------|--------|

5. **Individual Dataset Reports** — One section per domain:
   - Record count, variable count
   - Required Variables table (✓ PRESENT / ✗ MISSING)
   - Variable Detail table (Variable | Label | Type | Missing % | CT Issues | Severity)
   - Date Format findings (if `--DTC` variables present)
   - Controlled Terminology violations (Value | Count | Approved Values)
   - Cross-domain consistency result (SDTM only)
   - Domain verdict: **PASS** / **WARNING** / **BLOCKING**

6. **RAG Reference Log** — Table of every `mcp__npm-rag-v1__query_documents` call: query text and one-line result summary.

The QMD is written as a **static document** — all tables are pre-computed markdown, no R code blocks. It renders without re-running any code.

## Finding Format

```
**[SEVERITY]** `DOMAIN.VARIABLE` — description

- Actual: <what was found in the data>
- Expected: <what CDISC requires>
- CDISC Reference: <codelist ID or IG section from RAG>
- Suggested fix: <concrete action>
```
